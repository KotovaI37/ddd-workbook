Тактическое проектирование.

1\. Для выбора паттерна реализации будем использовать дерево тактических решений.

_Персонализация_ – _Транзакционный сценарий_. Алгоритмическая логика, нет сложных бизнес-правил, рекомендации могут быть конечной согласованности.

_Управление каталогом_ - _Активная запись_. Больше похоже на работу с БД, но есть инварианты (книга не может быть без автора). CRUD-операции с простыми инвариантами.

_Читалка_ - _Активная запись (клиент) + Модель предметной области (сервер)._ На сервере нужно отслеживать прогресс, но правила не сверхсложные.

_Биллинг_ - _Модель предметной области_. Сложные инварианты, финансовые ограничения, множество переходов состояний.

_DRM (Управление доступом)_ - _Модель предметной области, основанная на событиях_. Требуется полный аудит, возможность воспроизведения состояния на любой момент, юридические требования.

_Уведомления_ - _Транзакционный сценарий_. Простая логика отправки по триггерам.

_Пользователи_ - _Активная запись_. Типовые операции с учетными записями, можно использовать готовые решения.

2\. Проектирование тактической модели.

Спроектируем модель для двух bounded contexts.

1) _DRM (Управление доступом) — Event Sourced Domain Model_:

|     |     |
| --- | --- |
| Поле | Значение |
| Bounded Context | Управление доступом (DRM) |
| Агрегат (корень) | Лицензия (License) |
| Сущности внутри агрегата | Устройство (Device) — как часть агрегата (список устройств) |
| Объекты-значения | Идентификатор лицензии (LicenseId), Идентификатор пользователя (UserId), Идентификатор книги (BookId), Идентификатор устройства (DeviceId), Статус лицензии (LicenseStatus: ACTIVE, REVOKED, EXPIRED), Тип лицензии (LicenseType: PURCHASE, SUBSCRIPTION) |
| Инварианты | 1) Одна лицензия на одну книгу для одного пользователя при покупке. 2) Не более 5 устройств на одну лицензию. 3) Подписочная лицензия истекает через месяц. 4) Нельзя выдать лицензию повторно на ту же книгу при покупке. 5) При отзыве лицензии доступ теряется немедленно. |
| Команды | ВыдатьЛицензию, ЗарегистрироватьУстройство, УдалитьУстройство, ОтозватьЛицензию, ПродлитьЛицензию |
| Порождаемые события | ЛицензияВыдана, УстройствоЗарегистрировано, УстройствоУдалено, ЛицензияОтозвана, ЛицензияПродлена, ДостигнутЛимитУстройств |
| Границы транзакций | События атомарно сохраняются в Event Store. Чтение через проекции с конечной согласованностью. Снэпшоты каждые 100 событий для оптимизации восстановления. |
| Паттерн бизнес-логики | Модель предметной области, основанная на событиях (event-sourced domain model) |

_Модель восстановления состояния_.

Состояние агрегата «Лицензия» восстанавливается путем последовательного применения всех событий, произошедших с данной лицензией:

1) ЛицензияВыдана → создается запись с ID пользователя, ID книги, статусом ACTIVE, датой выдачи, типом лицензии. Для подписочной лицензии устанавливается дата истечения.

2) УстройствоЗарегистрировано → устройство добавляется в список зарегистрированных.

3) УстройствоУдалено → устройство удаляется из списка.

4) ЛицензияОтозвана → статус меняется на REVOKED.

5) ЛицензияПродлена → обновляется дата истечения.

6) ДостигнутЛимитУстройств → событие фиксирует попытку регистрации сверх лимита (не меняет состояние).

_EventStore и ограничения:_

_Хранилище_ . Отдельная БД для событий (EventStoreDB или таблица в PostgreSQL с JSON-полем для payload).

_Снэпшоты_. Сохраняются каждые 100 событий для быстрого восстановления без перебора всей истории.

_Риски_:

1) Снижение производительности при очень длинной истории.

2) Сложность запросов к данным (требуется CQRS).

3) Дополнительное место для хранения событий.

2) _Биллинг — Classical Domain Model._

|     |     |
| --- | --- |
| Поле | Значения |
| Bounded Context | Биллинг (Billing) |
| Агрегат (корень) | Подписка (Subscription) |
| Сущности внутри агрегата | Платеж (Payment), Счет (Invoice) |
| Объекты-значения | Денежная сумма (Money), Период (BillingPeriod: MONTHLY, YEARLY), Идентификатор пользователя (UserId), Статус подписки (SubscriptionStatus: TRIAL, ACTIVE, PAUSED, CANCELED, EXPIRED), Идентификатор платежа (PaymentId), Дата истечения (ExpiryDate) |
| Инварианты | 1) Подписка не может быть активной без успешного первого платежа. 2) Нельзя списать средства дважды за один период. 3) Смена тарифа возможна только при активной подписке. 4) При отмене подписки доступ сохраняется до конца оплаченного периода. 5) Пробный период может быть только один раз для одного пользователя. 6) Сумма платежа должна соответствовать тарифу. |
| Команды | СоздатьПодписку (с пробным периодом или без), АктивироватьПодписку, СписатьСредства, ПродлитьПодписку, СменитьТариф, ОтменитьПодписку, ПриостановитьПодписку, ВозобновитьПодписку |
| Порождаемые события | ПодпискаСоздана, ПодпискаАктивирована, СредстваСписаны, ПодпискаПродлена, ТарифИзменен, ПодпискаОтменена, ПодпискаПриостановлена, ПодпискаВозобновлена, ПодпискаИстекает (за N дней) |
| Границы транзакций | Все изменения внутри агрегата «Подписка» — одна атомарная транзакция. События публикуются после успешного завершения транзакции (outbox pattern). |
| Паттерн бизнес-логики | Модель предметной области (domain model) |

В контексте Биллинга требуется один доменный сервис:

PaymentGatewayService . Взаимодействие с внешним платежным шлюзом. Инкапсулирует логику вызова внешней системы, преобразования ответов, обработки ошибок. Не принадлежит ни одному агрегату, так как работа с внешними системами — отдельная ответственность.
