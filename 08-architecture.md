2\. _Интеграция между контекстами._

Для обеспечения надежности доставки событий между контекстами используем паттерн _Outbox_:

1) В рамках одной локальной транзакции агрегат сохраняет изменения в БД и публикует события в _outbox-таблицу_.

2) Отдельный процесс (publisher) читает события из outbox и отправляет их в брокер сообщений (Kafka/RabbitMQ).

3) После подтверждения доставки события помечаются как отправленные.

Опишем типы событий для интеграции:

1) _Событие предметной области (Domain Event)_: фиксирует факт, произошедший в агрегате. Реализуется с помощью LicenseIssued, PaymentCompleted. Применять это событие Для уведомления других контекстов о важных изменениях в ядре системы.

2) _Уведомление (Event Notification)_: сообщает, что что-то произошло, без передачи данных. Реализуется с помощью RecommendationsUpdated (только факт обновления). Применимо, когда потребитель сам может запросить данные или ему нужно только знать о событии.

3) Передача состояния через событие (Event-Carried State Transfer, ECST): событие содержит все данные, необходимые потребителю. Реализация за счет BookPublished с полными метаданными книги. Применяется, когда потребитель не должен делать дополнительный запрос; для уменьшения связности.

Теперь разграничим публичные (интеграционные) и приватные (внутренние) события:

_Публичные (интеграционные):_ BookPublished, LicenseIssued, PaymentCompleted. Публикуются в брокер сообщений, доступны всем подписчикам.

_Приватные (внутренние)_: OrderCreated внутри Биллинга (до платежа). Только внутри контекста, не публикуются в брокер.

3\. _Сквозной процесс между контекстами._

Рассмотрим сага (saga) vs. диспетчер процессов (process manager).

Анализирование сквозных процессов в домене «Электронная библиотека»:

1) _Покупка книги_. Участвуют в этом контексты: Читатель → Биллинг → DRM → Уведомления. Используемы тип процесса в этом событии Оркестрируемая сага (Orchestrated Saga).

2) _Регистрация + приветственная подписка_. Участвующие контексты: Пользователи → Биллинг → Уведомления. Хореографическая сага (Choreographed Saga).

3) _Публикация новой книги + уведомление подписчиков_. Участвующие контексты: Каталог → Персонализация → Уведомления. Диспетчер процессов (Process Manager).

_Выбор подхода или отказ._

Выбираем подход для процесса «Покупка книги».

1) Читатель инициирует покупку → Биллинг создает заказ;

2) Биллинг отправляет команду на платеж (внешняя система);

3) При успешном платеже Биллинг публикует PaymentCompleted;

4) DRM получает событие и выдает лицензию (LicenseIssued);

5) Уведомления получают LicenseIssued и отправляют письмо читателю.

Для этого процесса подойдет Оркестрируемая сага (Orchestrated Saga), т.к. есть четкий координатор – Биллинг, требуется компенсация при сбоях (если платеж не прошел, не выдавать лицензию), биллинг хранит состояние процесса (заказ, статус платежа).

Выбор подхода для процесса «Публикация новой книги»:

1) Администратор публикует новую книгу (BookPublished);

2) Нужно определить подписчиков на этого автора/жанр;

3) Сформировать персонализированные уведомления;

4) Отправить уведомления.

Подойдет Диспетчер процессов (Process Manager), т.к. процесс требует принятия решений (кто подписан на автора?), нужно собирать данные из разных источников (подписки пользователей), состояние процесса может храниться в отдельном компоненте.
